# 家庭食谱小程序

## 项目概述
这是一个家庭食谱管理小程序，旨在帮助家庭成员更好地规划和管理日常饮食。通过这个小程序，家庭成员可以共同规划未来一周的食谱，分享美食心得，提高家庭生活质量。

## 技术栈与项目结构
- 技术栈：Taro + React + TypeScript + Redux
- 目录结构：
```
src/
  ├── pages/          # 页面文件
  ├── components/     # 公共组件
  ├── stores/         # 状态管理
  ├── types/          # TypeScript 类型定义
  └── app.config.ts   # 应用配置
```

## 云函数
- 云函数职责：负责业务逻辑处理、数据存储和权限控制，不处理图片上传（由前端上传到云存储）。
- 调用方式：前端通过 Taro.cloud.callFunction 调用云函数，云函数返回数据或错误信息。
- 数据模型：云函数内部使用与前端一致的数据模型（如 Recipe、Family、DailyMenu），确保数据一致性。
- 权限控制：云函数内部进行权限校验，确保只有授权用户才能执行相应操作。

## UI/交互规范
- 所有有背景色的按钮（如页面主操作按钮、弹窗按钮等），必须统一使用 `$color-primary` 作为背景色，并保持白色文字，且加上点击动画。
- Loading 统一使用全局/局部 Loading 组件（`src/components/Loading`），支持全屏遮罩和局部加载，样式美观，避免在按钮上直接用 loading。
- 所有页面和组件样式均引用 SCSS 变量，便于主题维护，响应式设计全部用 rem 单位。

## 头像上传与资料编辑
- 头像上传逻辑统一封装为 `useAvatarUpload` hook，避免重复代码。调用时自动判断是否为 fileID，已是 fileID 则不再上传。
- 资料编辑页面（如昵称/头像）保存时，务必用临时变量保存 fileID，避免 setState 异步导致取到旧值。
- 头像上传示例：
  ```ts
  const fileID = await useAvatarUpload(avatar, user!.openId)
  await dispatch(updateUserInfo({ avatar, avatarFileId: fileID, ... }))
  ```

## 状态管理
- Redux Store 设计：userStore、familyStore、recipeStore 分模块管理。
- 异步 loading 状态推荐用 extraReducers 自动管理（如 fetchLoading、createLoading、joinLoading），每个异步操作独立 loading 字段。

## 其它重要实践
- 组件化开发，UI与逻辑分离，便于维护和复用。
- 所有页面主操作按钮、弹窗按钮等必须用 `$color-primary` 背景色，白色文字，带动画（如点击缩放/变色）。
- 避免 setState 后立即读取新 state，需用临时变量保证数据一致性。
- 所有云函数调用、异步副作用均用 thunk 封装，组件只 dispatch，不直接处理副作用。

# 模块化开发与维护攻略（AI专用）

## 1. 前端-用户中心（profile/edit）
- 头像选择：使用微信小程序 openType="chooseAvatar"，onChooseAvatar 事件获取微信头像本地临时路径。
- 头像本地路径需用 Taro.getFileSystemManager().readFile 读取为 base64，存入 avatar64 state，便于后续上传。
- 昵称输入框 type="nickname"，支持微信昵称自动填充。
- 保存时，将 avatar64（或 fileID，见下）和 nickname 一起传给后端云函数。
- 头像展示优先用本地 avatar 状态，用户可预览新头像。
- 选择头像、昵称后，前端不直接上传图片到云函数（腾讯云 CloudBase 场景），而是：
  - 前端用 Taro.cloud.uploadFile 上传图片到云存储，拿到 fileID。
  - 数据库只存 fileID，展示时用 fileID 转 HTTPS 链接。

## 2. 前端-Redux 状态管理
- user 模块 state 结构优化：
  - UserState.current: User | null
  - 其它字段如 isLoggedIn, code, loginLoading, profileLoading
- 所有 getState().user.user 改为 getState().user.current，selector 也同步。
- 登录 loading、资料 loading 分开管理（loginLoading/profileLoading）。
- 组件只 dispatch thunk，不直接处理副作用。
- 头像、昵称等本地 state 只做 UI 预览，最终以 redux/current 为准。

## 3. 后端-云函数/数据库
- 腾讯云 CloudBase 场景：
  - 后端云函数不负责图片上传，只负责存 fileID 到数据库。
  - 前端上传图片到云存储，云函数只做 update/add。
- 用户注册/登录云函数（wechat-login）：
  - 新用户 add 
  - 返回完整用户信息。
- 用户信息更新云函数（update-user-info）：
  - 接收 openId、nickName、avatarUrl（fileID），直接 update。
  - 不处理本地临时路径或 base64，前端需先上传。
- 数据库 user 表字段统一：openId、nickName、avatarUrl、createTime 等。
- 后台管理系统展示头像时，用 fileID 转 HTTPS 链接。

## 家庭模块

### 1. 数据结构
```typescript
// 家庭信息
interface Family {
  _id: string;
  name: string;
  family_owner: string;  // 家庭创建者openId
  members: string[];     // 成员openId列表
  membersInfo?: User[];  // 成员详细信息
  createdAt: Date;
  updatedAt: Date;
}

// 用户信息
interface User {
  _id: string;
  openId: string;
  nickname: string;
  avatar: string;
  family_id?: string;    // 所属家庭ID
  role: 'owner' | 'member';
  createdAt: number;
  updatedAt: number;
}
```

### 2. 创建家庭流程
1. 前端调用 `create-family` 云函数
2. 云函数处理：
   - 检查用户是否已有家庭
   - 创建新家庭记录，将创建者加入 members 数组
   - 更新用户表的 role 为 'owner' 和 family_id
3. 创建成功后：
   - 重新获取家庭信息（fetchFamily）
   - 显示 loading 状态
   - 成功后显示成功提示

### 3. 成员管理
- 创建者自动成为家庭管理员（role: 'owner'）
- 成员列表显示：
  - 创建者显示为"家庭管理员"
  - 其他成员显示为"成员"
  - 当前用户显示为"我"
- 成员信息通过 `get-user-info` 云函数获取

### 4. 状态管理
```typescript
// Redux Store
interface FamilyState {
  currentFamily: Family | null;
  membersInfo: User[];
  fetchLoading: boolean;
  createLoading: boolean;
  joinLoading: boolean;
}

// 异步操作
- fetchFamily: 获取家庭信息
- createFamily: 创建家庭
- joinFamily: 加入家庭
- leaveFamily: 退出家庭
```

### 5. 云函数说明
1. `create-family`：
   - 创建家庭记录
   - 更新用户角色和家庭ID
   - 返回完整的家庭信息

2. `get-family-info`：
   - 获取家庭基本信息
   - 调用 `get-user-info` 获取成员详细信息
   - 返回带成员信息的完整家庭数据

3. `get-user-info`：
   - 根据 openId 列表查询用户信息
   - 返回用户详细信息（昵称、头像等）

### 6. 注意事项
1. 创建家庭时必须更新用户的 family_id 和 role
2. 获取家庭信息时要同时获取成员详细信息
3. 所有异步操作都要显示 loading 状态
4. 成员列表要显示创建者信息
5. 用户角色分为 owner（家庭管理员）和 member（普通成员）

```typescript
interface Family {
  id: string;
  name: string;
  owner: string;       // 主人ID
  members: {           // 成员列表
    userId: string;
    role: 'owner' | 'member';
    permissions: string[];
  }[];
  createdAt: Date;
}
```

# 组件化与样式变量最佳实践

## 1. 组件拆分与迁移

- 组件通过 props 传递数据和事件，页面只负责数据流转和状态管理。

## 2. 样式变量化
- 所有颜色、圆角、阴影、间距等样式都要用变量（如 `$color-primary`、`$radius-lg` 等），不要写死颜色或尺寸。
- 如有新需求，优先在 `src/use/variables.scss` 中新增变量，保持全局风格一致。
- 组件样式文件如需用到主色、文本色、圆角等，直接用变量名。

## 3. SCSS @use 规范
- **每个组件/样式文件、页面文件都要在顶部加：**
  ```scss
  @use '../../use/variables' as *; // ⭐注意路径要对！页面和组件不同层级要写对
  @use "sass:color"; // ⭐必须加，避免色彩函数报错
  ```

- `@use '../../use/variables' as *;` 用于引入全局变量。
- `@use "sass:color";` 用于支持 `color.scale` 等新Sass颜色函数，避免 `lighten` 等废弃警告。
- **每个页面/组件的 index.tsx/tsx 都要 import './index.scss'，否则样式不生效。**

## 4. 变量用法示例
- 颜色：`color: $color-text-main;`、`background: $color-member-bg;`、`border: 2px solid color.scale($color-primary, $lightness: 40%);`

- 圆角：`border-radius: $radius-lg;`
- 按钮等交互色：`background: $color-primary; color: $color-bg-card;`

## 5. 组件复用与类型安全

## React 动画最佳实践

### 1. 动画方向处理
在处理需要根据用户操作改变方向的动画时（如滑动切换），需要注意状态更新的时序问题。以下是推荐的实现方式：

```typescript
const Component = () => {
  // 当前显示的数据
  const [selectedDate, setSelectedDate] = useState(new Date())
  // 待更新的数据
  const [pendingDate, setPendingDate] = useState<Date | null>(null)
  // 动画方向
  const [slideDir, setSlideDir] = useState<'left' | 'right'>('left')

  // 使用 useLayoutEffect 确保方向先于数据更新
  useLayoutEffect(() => {
    if (pendingDate) {
      setSelectedDate(pendingDate)
      setPendingDate(null)
    }
  }, [pendingDate])

  // 统一的数据更新函数
  const updateDate = (newDate: Date) => {
    // 先设置动画方向
    if (newDate > selectedDate) {
      setSlideDir('left')
    } else if (newDate < selectedDate) {
      setSlideDir('right')
    }
    // 再设置待更新数据
    setPendingDate(newDate)
  }

  return (
    <SwitchTransition mode="out-in">
      <CSSTransition
        key={getDateKey(selectedDate)}
        timeout={300}
        classNames={slideDir === 'left' ? 'slide-left' : 'slide-right'}
      >
        <View>
          {/* 内容 */}
        </View>
      </CSSTransition>
    </SwitchTransition>
  )
}
```

关键点说明：
1. 使用 `pendingDate` 状态来暂存待更新的数据，避免直接更新 `selectedDate`
2. 使用 `useLayoutEffect` 确保动画方向在 DOM 更新前就设置好
3. 在 `updateDate` 函数中先设置方向，再设置待更新数据
4. CSSTransition 的 key 只需要使用数据标识，不需要包含方向信息

这种方式的优点：
- 避免了状态更新时序问题
- 确保动画方向在 DOM 更新前就准备好
- 代码结构清晰，易于维护
- 可以处理连续快速切换的情况

# 项目主要功能与实现（近期改动）

## 1. 登录持久化
- 登录成功后，`token` 和 `user` 信息自动存入本地 storage。

- `callCloud` 工具自动从本地获取 token 并带入每次云函数调用。
- 云函数返回统一格式（code/message/data），非 code=0 自动抛异常。
- 前端所有调用无需再手动判断 code，异常自动处理。

## 3. 全局 Loading 处理
- 新增 `AppContainer` 组件，负责全局 Loading 展示。
- 通过 Redux selectors 获取 family/recipe 的 loading 状态，只要有一个在加载即显示全局 Loading。
- Loading 组件见 `src/components/Loading`。

- App 启动且已登录时，自动拉取家庭（family）和菜谱（recipes）数据。
- 统一初始化逻辑封装在 `initApp` thunk（`src/thunks/initApp.ts`）。
- 初始化流程完成前，Loading 全局可见。
