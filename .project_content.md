# 家庭食谱小程序

## 项目概述
这是一个家庭食谱管理小程序，旨在帮助家庭成员更好地规划和管理日常饮食。通过这个小程序，家庭成员可以共同规划未来一周的食谱，分享美食心得，提高家庭生活质量。

## 技术栈与项目结构
- 技术栈：Taro + React + TypeScript + MobX
- 目录结构：
```
src/
  ├── pages/          # 页面文件
  ├── components/     # 公共组件
  ├── stores/         # 状态管理
  ├── types/          # TypeScript 类型定义
  └── app.config.ts   # 应用配置
```

# 模块化开发与维护攻略（AI专用）

## 1. 前端-用户中心（profile/edit）
- 头像选择：使用微信小程序 openType="chooseAvatar"，onChooseAvatar 事件获取微信头像本地临时路径。
- 头像本地路径需用 Taro.getFileSystemManager().readFile 读取为 base64，存入 avatar64 state，便于后续上传。
- 昵称输入框 type="nickname"，支持微信昵称自动填充。
- 保存时，将 avatar64（或 fileID，见下）和 nickname 一起传给后端云函数。
- 头像展示优先用本地 avatar 状态，用户可预览新头像。
- 选择头像、昵称后，前端不直接上传图片到云函数（腾讯云 CloudBase 场景），而是：
  - 前端用 Taro.cloud.uploadFile 上传图片到云存储，拿到 fileID。
  - 数据库只存 fileID，展示时用 fileID 转 HTTPS 链接。

## 2. 前端-Redux 状态管理
- user 模块 state 结构优化：
  - UserState.current: User | null
  - 其它字段如 isLoggedIn, code, loginLoading, profileLoading
- 所有 getState().user.user 改为 getState().user.current，selector 也同步。
- 登录 loading、资料 loading 分开管理（loginLoading/profileLoading）。
- 组件只 dispatch thunk，不直接处理副作用。
- 头像、昵称等本地 state 只做 UI 预览，最终以 redux/current 为准。

## 3. 后端-云函数/数据库
- 腾讯云 CloudBase 场景：
  - 后端云函数不负责图片上传，只负责存 fileID 到数据库。
  - 前端上传图片到云存储，云函数只做 update/add。
- 用户注册/登录云函数（wechat-login）：
  - 新用户 add 
  - 返回完整用户信息。
- 用户信息更新云函数（update-user-info）：
  - 接收 openId、nickName、avatarUrl（fileID），直接 update。
  - 不处理本地临时路径或 base64，前端需先上传。
- 数据库 user 表字段统一：openId、nickName、avatarUrl、createTime 等。
- 后台管理系统展示头像时，用 fileID 转 HTTPS 链接。

# 其它重要实践
- 所有异步副作用（如云函数调用、图片上传）都应放在 thunk 里，组件只 dispatch。
- 组件 UI 状态（如 loading）全部用 redux 控制。
- 头像、昵称等字段变更后，redux/current 需同步更新。
- 数据库安全规则、字段名需前后端一致。
